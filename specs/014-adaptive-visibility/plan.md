# Implementation Plan: Adaptive Dashboard Visibility (On-Track Only)

**Branch**: `014-adaptive-visibility` | **Date**: 2026-04-12 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `/specs/014-adaptive-visibility/spec.md`

---

## Summary

Add two boolean fields (`isOnTrack`, `isReplayPlaying`) to the existing `TelemetrySnapshot` domain type. Map them from the iRacing SDK (`IsOnTrack`, `IsReplayPlaying`) in the live adapter and from scenario flags in the mock adapter. In the renderer, evaluate `isVisible = frame !== null && frame.snapshot.isOnTrack && !frame.snapshot.isReplayPlaying` and apply `visibility: hidden` (CSS) to the top-level overlay container when false. No new IPC channels, no window property changes, no state machines.

## Specialist Consultation

- [x] **Telemetry Impact?** — @telemetry-expert consulted.
  - **Finding**: Use `IsOnTrack` (player-scoped boolean, SDK line 260) and `IsReplayPlaying` (line 270). `SessionState` is not needed as a secondary gate — `IsOnTrack` alone correctly handles all garage/menu/pit scenarios.
  - **Ghost Prevention**: `IsOnTrack` goes `false` immediately when the player presses Escape and the car is teleported to the garage. No debounce or secondary check needed.

- [x] **UI/UX Changes?** — @ui-stylist consulted.
  - **Finding**: Use `visibility: hidden` (not `display: none`, not `opacity: 0`, not `BrowserWindow.hide()`). This preserves React component state and avoids DWM recomposition. Zero mouse-interception is already guaranteed by the existing `setIgnoreMouseEvents(true)` lock mode.

## Implementation Steps

### Step 1 — Domain: Extend `TelemetrySnapshot`

**File**: `src/domain/telemetry/types.ts`

Add two fields to the `TelemetrySnapshot` interface:

```typescript
isOnTrack: boolean;
isReplayPlaying: boolean;
```

These are non-nullable booleans with an implicit default of `false` when no data is available.

---

### Step 2 — Adapter (Live): Map SDK Variables

**File**: `src/adapters/telemetry-iracing/iracing-telemetry-provider.ts`

In the `poll()` method, read the two new SDK variables from `getTelemetry()` and include them in the `this.latest` object:

```typescript
// Inside try block, after const t = this.sdk.getTelemetry();
const isOnTrack = t.IsOnTrack?.value?.[0] ?? false;
const isReplayPlaying = t.IsReplayPlaying?.value?.[0] ?? false;

// Add to this.latest = { ... }:
isOnTrack,
isReplayPlaying,
```

**Performance note**: Both are simple array index reads on an already-fetched telemetry object. Zero additional SDK calls.

---

### Step 3 — Adapter (Mock): Add Default + New Scenarios

**File**: `src/adapters/telemetry-mock/mock-telemetry-provider.ts`

- Set `isOnTrack: true` and `isReplayPlaying: false` in the default/existing mock snapshot outputs (cars on track).
- Add a `garage` mock scenario: `isOnTrack: false`, `isReplayPlaying: false`.
- Add a `replay` mock scenario: `isOnTrack: true`, `isReplayPlaying: true`.

This enables validation with:
```
POSHDASH_USE_MOCK=1 POSHDASH_MOCK_SCENARIO=garage npm run dev   # overlay should be invisible
POSHDASH_USE_MOCK=1 POSHDASH_MOCK_SCENARIO=replay npm run dev   # overlay should be invisible
```

---

### Step 4 — Renderer: Derive and Apply Visibility

**File**: `src/renderer/App.tsx`

Derive `isVisible` from the telemetry frame and apply it to the `<Overlay>` container wrapper:

```tsx
const isVisible = frame !== null && frame.snapshot.isOnTrack && !frame.snapshot.isReplayPlaying;

return (
  <div style={{ visibility: isVisible ? 'visible' : 'hidden' }}>
    <Overlay frame={frame} waitingMessage={waitingMsg} locked={locked} />
  </div>
);
```

**Why a wrapper div?** The `<Overlay>` component manages its own layout and styles. Adding the `visibility` style to a neutral wrapper avoids touching Overlay's internal styles and keeps concerns clean.

**No `pointer-events: none`** needed — the window is already `setIgnoreMouseEvents(true)` in locked mode.
